# Отчёт по лабораторной работе №1

## «Применение паттерна "Фбричный метод" в игре Tower Defense на Unity»

### 1. Введение

Цель работы – изучение и практическое применение паттерна проектирования «Фабричный метод» для создания врагов в игре Tower Defense. В игре присутствуют различные типы врагов (обычный, быстрый, танк), которые создаются волнами. Требуется организовать создание объектов так, чтобы добавление новых типов не требовало изменений в существующем коде спавнера.

### 2. Анализ реализации без применения паттерна

В исходном коде (предоставленном ранее) подсистема врагов отсутствовала, однако типичная реализация без паттерна выглядит следующим образом:

- Каждый тип врага представлен отдельным префабом.
- Спавнер содержит ссылки на все префабы и при создании выбирает нужный через условный оператор (switch или if).

```csharp
public class EnemySpawner : MonoBehaviour
{
    public GameObject ordinaryPrefab;
    public GameObject fastPrefab;
    public GameObject tankPrefab;

    void SpawnEnemy(EnemyType type)
    {
        GameObject prefab = null;
        switch (type)
        {
            case EnemyType.Ordinary: prefab = ordinaryPrefab; break;
            case EnemyType.Fast: prefab = fastPrefab; break;
            case EnemyType.Tank: prefab = tankPrefab; break;
        }
        Instantiate(prefab, spawnPoint.position, Quaternion.identity);
    }
}
```

**Недостатки:**

- Жёсткая связность: спавнер зависит от конкретных префабов и перечисления типов.
- Нарушение OCP: добавление нового врага требует изменения класса спавнера (новая ветка switch, новое поле).
- Дублирование кода создания при необходимости дополнительной настройки врага.
- Сложность тестирования и модификации.

### 3. Паттерн «Фабричный метод»

**Назначение:** определяет интерфейс создания объекта, но позволяет подклассам изменять тип создаваемого объекта. Создание делегируется наследникам.

### 4. Применение паттерна в проекте

В предоставленной реализации (последние файлы) паттерн применён следующим образом:

- **Абстрактный продукт** – класс `Enemy` (содержит общие характеристики и методы).
- **Конкретные продукты** – `EnemyOrdinary`, `EnemyFast`, `EnemyTank` (наследуют `Enemy` и реализуют специфическое поведение).
- **Абстрактный создатель** – класс `EnemyFactory` с фабричным методом `CreateEnemy(Vector3 position)`.
- **Конкретные создатели** – `OrdinaryEnemyFactory`, `FastEnemyFactory`, `TankEnemyFactory`. Каждая фабрика хранит ссылку на свой префаб и в методе `CreateEnemy` создаёт экземпляр соответствующего врага.
- **Клиент** – класс `EnemySpawner` получает массив фабрик через инспектор и при спавне выбирает случайную фабрику и вызывает её метод `CreateEnemy`.

Код спавнера стал предельно простым:

```csharp
public class EnemySpawner : MonoBehaviour
{
    [SerializeField] private EnemyFactory[] enemyFactories;

    private void SpawnEnemy()
    {
        EnemyFactory factory = enemyFactories[Random.Range(0, enemyFactories.Length)];
        factory.CreateEnemy(LevelManager.main.startPoint.position);
    }
}
```

### 5. Как паттерн решает проблему

- **Инкапсуляция создания**: логика создания каждого типа врага сосредоточена в своей фабрике.
- **Лёгкость расширения**: новый тип врага добавляется созданием нового класса врага и новой фабрики. Достаточно включить фабрику в массив спавнера – никаких правок в существующем коде не требуется.
- **Соблюдение OCP**: классы открыты для расширения, но закрыты для модификации.
- **Единообразие**: клиент работает с абстрактными `Enemy` и `EnemyFactory`, не завися от конкретных реализаций.
- **Упрощение тестирования**: можно подменить фабрику на mock и тестировать логику спавна изолированно.

Таким образом, паттерн «Фабричный метод» успешно применён для создания врагов, обеспечивая гибкость и расширяемость системы.
