# Отчёт по лабораторной работе №1

## «Применение паттерна "Абстрактная фабрика" в игре Tower Defense на Unity»

### 1. Введение

Целью лабораторной работы является изучение и практическое применение паттерна проектирования **«Абстрактная фабрика»** (Abstract Factory) при разработке компьютерной игры в жанре Tower Defense на движке Unity.

В игре присутствуют следующие ключевые сущности:

- **Башни** – оборонительные сооружения, которые игрок размещает на карте для уничтожения врагов.
- **Враги** – перемещаются по маршруту, башни атакуют их.
- **Уровни сложности / волны** – определяют количество и типы врагов.

Работа фокусируется на проектировании подсистемы создания башен. Башни бывают разных **типов** (например, турель, замораживающая пушка Tesla) и должны иметь несколько **уровней прокачки** (1–3). Каждый уровень изменяет внешний вид, характеристики и поведение башни (последнее реализовано не в полной мере). Задача – организовать создание объектов башен так, чтобы система легко расширялась новыми типами и уровнями без модификации существующего клиентского кода.

В отчёте представлен анализ исходной реализации (без применения паттерна), описание паттерна «Абстрактная фабрика» и его реализация в разработанном проекте, а также сравнение двух подходов.

### 2. Анализ исходной реализации (без применения паттерна)

В исходных кодах решения без паттерна (файлы: `BuildManager.cs`, `Plot.cs`, `Tesla.cs`, `Turret.cs`, `Tower.cs`) реализована базовая механика размещения башен без использования паттернов проектирования. Рассмотрим ключевые классы.

#### 2.1. Класс `Tower`

Простая модель данных, хранящая имя, стоимость и префаб башни:

```csharp
[Serializable]
public class Tower
{
    public string name;
    public int cost;
    public GameObject prefab;

    public Tower(string name, int cost, GameObject prefab)
    {
        this.name = name;
        this.cost = cost;
        this.prefab = prefab;
    }
}
```

#### 2.2. Класс `BuildManager`

Управляет выбором текущей башни для постройки. Хранит массив доступных башен (`Tower[] towers`) и индекс выбранной.

```csharp
public class BuildManager : MonoBehaviour
{
    public static BuildManager main;

    [Header("References")]
    [SerializeField] private Tower[] towers;
    [SerializeField] private int selectedTower = 0;

    private void Awake() { main = this; }

    public Tower GetSelectedTower() { return towers[selectedTower]; }
    public void SetSelectedTower(int _selectedTower) { selectedTower = _selectedTower; }
}
```

#### 2.3. Класс `Plot`

Обрабатывает клик по клетке для постройки башни. При нажатии проверяет, не занято ли место, достаточно ли валюты, затем создаёт экземпляр префаба выбранной башни.

```csharp
private void OnMouseDown()
{
    if (tower != null) return;

    Tower towerToBuild = BuildManager.main.GetSelectedTower();

    if (towerToBuild.cost > LevelManager.main.currency)
    {
        Debug.Log("Not enough currency to build this tower");
        return;
    }

    LevelManager.main.DecreaseCurrency(towerToBuild.cost);
    tower = Instantiate(towerToBuild.prefab, transform.position, Quaternion.identity);
}
```

#### 2.4. Конкретные башни (`Turret` и `Tesla`)

Два типа башен реализованы как отдельные MonoBehaviour-скрипты. Они не имеют общего предка (кроме MonoBehaviour) и не реализуют общий интерфейс.

**Turret** – стреляющая башня, вращается в сторону цели и выпускает снаряды.

**Tesla** – замораживающая башня, воздействует на всех врагов в радиусе, замедляя их и нанося урон.

#### 2.5. Недостатки текущего подхода

1. **Отсутствие абстракции для башен**. Нет общего интерфейса или базового класса, что затрудняет единообразную работу с разными типами башен (например, вызов метода `Attack()`).
2. **Жёсткая связность создания объекта**. Код создания (`Instantiate`) находится прямо в `Plot` и использует конкретный префаб из `Tower`. Любое изменение в логике создания потребует правки `Plot`.
3. **Сложность добавления новых типов и уровней**. Чтобы добавить новую башню, нужно:
   - Создать новый скрипт (или использовать существующий, если поведение аналогично).
   - Добавить префаб в массив `towers` в `BuildManager`.
   - При необходимости модифицировать `Plot` или другие классы, если у новой башни есть особые требования.
   - Для поддержки уровней пришлось бы заводить отдельные префабы для каждого уровня и добавлять их все в массив, что приведёт к разрастанию массива и усложнению выбора.
4. **Нарушение принципа открытости/закрытости (OCP)**. Добавление новой башни требует изменения класса `BuildManager` (добавление элемента в массив) и, возможно, `Plot`.
5. **Дублирование кода**. Логика постройки (проверка валюты, инстанцирование) находится в `Plot` и будет повторяться для всех типов построек.
6. **Трудность тестирования**. Невозможно подменить создание башни на mock-объект для юнит-тестов.

### 3. Паттерн «Абстрактная фабрика»

**Назначение:** предоставляет интерфейс для создания семейств взаимосвязанных или взаимозависимых объектов, не специфицируя их конкретные классы.

**Структура паттерна** (по GoF):

- **AbstractFactory** – абстрактный интерфейс с методами создания абстрактных продуктов.
- **ConcreteFactory** – реализует методы создания конкретных продуктов одного семейства.
- **AbstractProduct** – абстрактный интерфейс для типа продукта.
- **ConcreteProduct** – конкретная реализация продукта, создаваемая соответствующей фабрикой.
- **Client** – использует только интерфейсы AbstractFactory и AbstractProduct.

В классическом варианте для каждого семейства продуктов создаётся своя конкретная фабрика. В нашем случае семействами являются **уровни башен**, а продуктами – **типы башен** (турель, Tesla и т.д.).

### 4. Применение паттерна в разработанном решении

В исходных файлах решения с паттерном (после рефакторинга) реализована архитектура, соответствующая паттерну «Абстрактная фабрика». Её ключевые компоненты.

#### 4.1. Абстрактные продукты

Были созданы два абстрактных класса – `Turret` и `Tesla`. Они определяют общее поведение для всех башен соответствующего типа, но оставляют возможность для переопределения конкретных методов в наследниках (уровнях).

**Класс `Turret` (абстрактный)**  
Содержит общую логику поиска цели, поворота, стрельбы. Метод `Shoot()` объявлен как `protected virtual`, что позволяет переопределить его в классах-наследниках для изменения поведения стрельбы на разных уровнях.

**Класс `Tesla` (абстрактный)**  
Содержит базовую логику замораживания и нанесения урона. Методы `PerformEffect()`, `ApplyDamage()`, `ApplyFreeze()` помечены как `protected virtual`, что даёт возможность изменять эффект на конкретных уровнях (например, `FirstLevelTesla` переопределяет `ApplyFreeze`, чтобы не замораживать врагов).

#### 4.2. Конкретные продукты

Для каждого уровня созданы свои классы, наследующие соответствующий абстрактный класс:

- **Turret**: `FirstLevelTurret`, `SecondLevelTurret`, `ThirdLevelTurret`
- **Tesla**: `FirstLevelTesla`, `SecondLevelTesla`, `ThirdLevelTesla`

Каждый из этих классов может переопределять виртуальные методы для реализации уникального поведения данного уровня. Например, `SecondLevelTurret` и `ThirdLevelTurret` переопределяет `Shoot()` для добавления спецэффектов (не реализовано, оставлены комментарии), а `FirstLevelTesla` отключает заморозку.

#### 4.3. Интерфейс абстрактной фабрики

В роли абстрактной фабрики выступает класс `AbstractFactory`:

```csharp
public abstract class AbstractFactory : MonoBehaviour
{
    public abstract Turret CreateTurret(Vector3 position);
    public abstract Tesla CreateTesla(Vector3 position);
    public abstract int TurretCost { get; }
    public abstract int TeslaCost { get; }
}
```

Он определяет методы создания продуктов обоих типов, а также свойства для получения стоимости (это упрощает проверку валюты до создания объекта).

#### 4.4. Конкретные фабрики

Для каждого уровня создана своя конкретная фабрика: `FirstLevelFactory`, `SecondLevelFactory`, `ThirdLevelFactory`. Каждая фабрика:

- Содержит ссылки на префабы башен соответствующего уровня (например, `turretPrefab` для `FirstLevelTurret`).
- Реализует методы создания, возвращая компонент нужного типа после инстанцирования префаба.
- Задаёт стоимости башен данного уровня.

Пример (`FirstLevelFactory`):

```csharp
public class FirstLevelFactory : AbstractFactory
{
    public GameObject turretPrefab;  // должен иметь компонент FirstLevelTurret
    public GameObject teslaPrefab;   // должен иметь компонент FirstLevelTesla

    public override int TurretCost => turretCost;
    public override int TeslaCost => teslaCost;

    public override Turret CreateTurret(Vector3 position)
    {
        GameObject go = Instantiate(turretPrefab, position, Quaternion.identity);
        return go.GetComponent<Turret>();
    }

    public override Tesla CreateTesla(Vector3 position)
    {
        GameObject go = Instantiate(teslaPrefab, position, Quaternion.identity);
        return go.GetComponent<Tesla>();
    }
}
```

#### 4.5. Клиент – класс `BuildManager`

`BuildManager` выступает в роли клиента, который использует фабрики. Он хранит ссылки на все три конкретные фабрики (назначаются в инспекторе), текущий выбранный тип башни (`TowerType`) и уровень. В методе `Update()` он кэширует актуальную фабрику в зависимости от выбранного уровня.

Основной метод `BuildTower(Vector3 position)`:

1. Проверяет, что фабрика существует и достаточно валюты.
2. Списывает валюту через `LevelManager.main.DecreaseCurrency(cost)`.
3. Вызывает соответствующий метод фабрики (`CreateTurret` или `CreateTesla`) и возвращает созданный `GameObject`.

Таким образом, `BuildManager` полностью абстрагирован от конкретных классов башен – он работает только с интерфейсами `AbstractFactory`, `Turret` и `Tesla`.

#### 4.6. Клиент – класс `Plot`

Класс `Plot` (клетка для постройки) предельно прост: при клике он вызывает `BuildManager.main.BuildTower(transform.position)` и сохраняет ссылку на созданный объект. Вся логика проверки валюты, выбора префаба и инстанцирования делегирована `BuildManager` и фабрикам.

```csharp
private void OnMouseDown()
{
    if (tower != null) return;
    tower = BuildManager.main.BuildTower(transform.position);
}
```

#### 4.7. Меню выбора (`Menu`)

Для взаимодействия с игроком добавлен класс `Menu`, который обрабатывает нажатия кнопок выбора типа башни и уровня. Он вызывает соответствующие методы `BuildManager` (`SelectTowerType` и `SelectLevel`) и визуально подсвечивает выбранные кнопки. Это позволяет динамически менять текущую конфигурацию без изменения кода.

#### 4.8. Диаграмма классов

На приложенной диаграмме (файл `OOAD-2_Lab-1.drawio1.xml`) показана структура, описанная выше. Основные участники:

- `AbstractFactory` – абстрактная фабрика.
- `FirstLevelFactory`, `SecondLevelFactory`, `ThirdLevelFactory` – конкретные фабрики.
- `Turret` (abstract) и `Tesla` (abstract) – абстрактные продукты.
- `FirstLevelTurret`, `SecondLevelTurret`, `ThirdLevelTurret` – конкретные продукты для турелей.
- `FirstLevelTesla`, `SecondLevelTesla`, `ThirdLevelTesla` – конкретные продукты для Tesla.
- `BuildManager` – клиент, использующий фабрики.
- `Plot` – клиент, использующий `BuildManager`.

Диаграмма наглядно демонстрирует, как паттерн отделяет клиента от конкретных классов продуктов и позволяет легко добавлять новые уровни или типы башен.

### 5. Как паттерн решает поставленные задачи

- **Инкапсуляция создания объектов**. Логика создания башен каждого уровня полностью скрыта внутри соответствующих фабрик. Клиент (`BuildManager` и `Plot`) не знает о существовании конкретных префабов и классов.
- **Лёгкость добавления новых типов башен**. Чтобы добавить новый тип (например, лучника), потребуется:
  1. Создать абстрактный класс `Archer`.
  2. Создать конкретные классы для каждого уровня (`FirstLevelArcher`, `SecondLevelArcher`, ...).
  3. Добавить в `AbstractFactory` методы создания нового типа (или расширить существующие фабрики новыми методами, если тип добавляется во все уровни).
  4. Создать новые фабрики или модифицировать существующие, чтобы они возвращали экземпляры нового типа.
  5. Добавить соответствующий пункт в меню и в `BuildManager` (новое значение `TowerType`).
  При этом существующий код (`Plot`, `BuildManager`) останется неизменным, кроме добавления новой ветви в `switch` для получения стоимости и создания.
- **Поддержка уровней**. Уровни явно представлены отдельными фабриками. Если потребуется добавить четвёртый уровень, достаточно создать `FourthLevelFactory` и соответствующие продукты. Никаких изменений в существующих фабриках не требуется.
- **Единообразие работы с башнями**. Все башни одного типа наследуют общий абстрактный класс, поэтому `BuildManager` может вызывать общие методы (например, через `GetComponent<Turret>()`), не заботясь о конкретном уровне. В перспективе можно добавить интерфейс `ITower` с методами `Attack()`, `Upgrade()` и т.д.
- **Соблюдение принципа открытости/закрытости (OCP)**. Классы фабрик и продуктов открыты для расширения (новые уровни, новые типы), но закрыты для модификации. Добавление новой функциональности не требует правки уже отлаженного кода.
- **Упрощение тестирования**. Можно создать mock-фабрику, возвращающую тестовые башни, и проверить логику `BuildManager` и `Plot` без реального инстанцирования префабов.
